**[ПОКАЗАТИ]** Дерево файлів: `pages/`, `tests/`, `utils/`, `conftest.py`, `.github/workflows/ci.yml`.

**[СКАЗАТИ]**

класу на кожну сторінку сайту. У папці `tests` — по одному файлу на кожен тест-
кейс, від `test_tc01` до `test_tc26`. У папці `utils` — генератор тестових даних

`ci.yml` — це CI-пайплайн. Ключова ідея — Page Object Model: логіка тесту
відокремлена від локаторів сторінки. Якщо змінюється інтерфейс, я правлю тільки
одне місце.»

## 1:05 — BasePage: головний патерн (50с)

**[ПОКАЗАТИ]** Відкрити `pages/base_page.py`. Повільно прокрутити `click`, `fill`, `expect_visible`.

**[СКАЗАТИ]**
«Це найважливіший клас — `BasePage`. Кожен Page Object успадковується від нього.
Я не викликаю Playwright напряму. Замість цього я обгортаю кожну дію — `click`,
`fill`, `check`, `expect_visible` — у метод-помічник. Кожна обгортка робить три
речі: відкриває Allure-крок із чітким описом, виконує дію Playwright і прикріплює
скриншот. Тому кожен крок у кожному тесті автоматично отримує опис і скриншот у
звіті — я не дублюю цей код у кожному тесті. Описи параметризовані: я передаю
текст як аргумент. Для перевірок я використовую `expect` від Playwright — він має
автоматичне очікування, тому тести стабільні й не флакі.»

**[ПОКАЗАТИ]** На мить відкрити `pages/login_page.py`.

**[СКАЗАТИ]**
«Ось реальний Page Object — сторінка логіну. У конструкторі я оголошую всі
локатори, використовуючи стабільні атрибути `data-qa`. Далі виношу дії, як-от
`login` і `signup`, і методи-перевірки, як `expect_login_error_visible`. Тести
ніколи не бачать сирий селектор — вони викликають лише ці іменовані методи.»

## 1:55 — Розбір одного тесту (35с)

**[ПОКАЗАТИ]** Відкрити `tests/test_tc02_login_valid.py`.

**[СКАЗАТИ]**
«Це Тест-кейс 2 — логін із коректними email і паролем. Зверху видно Allure-
декоратори: feature, story, severity і читабельний title. Тест читається як

`existing_user` — це фікстура: вона створює новий акаунт перед тестом і прибирає
його після. Тому кожен тест незалежний і повторюваний.»

## 2:30 — conftest.py: фікстури, дані, прибирання (35с)

**[ПОКАЗАТИ]** Відкрити `conftest.py`. Показати `block_ad_requests`, `registered_user`, `existing_user`, `pytest_runtest_makereport`.

**[СКАЗАТИ]**
«Уся спільна підготовка — у `conftest.py`. Я маю фікстуру на кожен Page Object,
тож тест просто просить те, що йому потрібно. Функція `generate_user` через Faker
створює унікальні тестові дані, тому тести реєстрації ніколи не конфліктують.
Фікстури `registered_user` і `existing_user` гарантують прибирання — вони
видаляють акаунт через API у teardown, навіть якщо тест впав раніше. Є ще autouse-
фікстура `block_ad_requests`, яка блокує рекламні запити, а цей хук,
`pytest_runtest_makereport`, прикріплює повний скриншот до Allure, коли тест
падає.»

## 3:05 — Запуск: паралельно і кросбраузерно (25с)

**[ПОКАЗАТИ]** Термінал із готовими командами:
```
pytest                         # за замовчуванням chromium
pytest -n auto                 # паралельно на всіх ядрах
pytest --browser firefox       # інший браузер
pytest --browser webkit
```

**[СКАЗАТИ]**
«Щоб запустити набір, я просто пишу `pytest`. З `-n auto` тести йдуть паралельно
на всіх ядрах процесора через pytest-xdist. Браузер обираю CLI-аргументом —
`--browser firefox` або `--browser webkit` — це дає плагін pytest-playwright. Ті
самі тести працюють на Chromium, Firefox і WebKit без змін у коді.»

## 3:50 — CI-пайплайн, GitHub Pages, Slack (30с)

**[ПОКАЗАТИ]** Відкрити `.github/workflows/ci.yml`, потім вкладку Actions, потім живий звіт на GitHub Pages, потім Slack-канал із повідомленням.

**[СКАЗАТИ]**
«Пайплайн на кожен push запускає matrix із трьох job, по одному на браузер,
паралельно. Job зі звітом об'єднує результати, будує Allure-звіт з історією і
деплоїть його на GitHub Pages — ось живий звіт. Останній job надсилає повідомлення
в Slack: кількість passed і failed, гілку і пряме посилання на звіт. Повний цикл:
залив код — тести прогнались на трьох браузерах — звіт опублікувався — команда
отримала сповіщення.»

## 4:20 — Реальний дебаг: два runtime-баги (30с)

**[ПОКАЗАТИ]** Відкрити `pages/cart_page.py` (рядки з `proceed_to_checkout_button` і коментарем поруч), потім `pages/products_page.py` (`page_heading`).

**[СКАЗАТИ]**
«Коли я прогнав увесь набір, два локатори падали не через помилку в моєму коді, а
через нюанс розмітки сайту. Тут, у `cart_page.py`, кнопка 'Proceed To Checkout' —
це посилання без атрибута `href`, тому Playwright не дає йому ролі `link` в
accessibility-дереві, і пошук за роллю просто не знаходив елемент. Виправлення —
шукати за класом `.check_out`. А тут, у `products_page.py`, заголовок секції
товарів ловив локатор, який також захоплював ціну кожного товару — там теж
`<h2>`. Звузив до `.title`. Це не вигадана демонстрація — я справді знайшов і
виправив ці два падіння через реальний прогін проти живого сайту.»
